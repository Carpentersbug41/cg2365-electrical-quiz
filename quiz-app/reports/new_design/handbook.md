# Guided Chunk Platform Handbook

## Purpose

This is the main concept document for the guided chunk platform. Read this first if you need to understand what the system is, why it exists, and how the core parts fit together.

## Problem

The current app already has strong curriculum grounding, retrieval practice, diagnostics, mastery logic, and AI generation. The main weakness is that the learning runtime can still feel assembled and static compared with the more dynamic onboarding experience. The goal of the new platform is to make lesson delivery feel alive without sacrificing pedagogic structure, curriculum control, or measurement quality.

A later product finding sharpened this further: a simple prompt-based tutor loop can already feel better than the current app runtime because it keeps teaching short, interaction immediate, and the tutor voice continuous. That benchmark is captured in `prompt_pattern_to_superior_product.md` and should shape how `guided_chunk_v1` is implemented.

The latest rebuild decision goes further: chunks should be treated as instructional briefs for the runtime model rather than final learner-facing text. That simpler direction is now captured in `dynamic_guided_runtime_v2.md` and should override older over-engineered guided-runtime assumptions where they conflict.

There is now also a hard implementation boundary: the rebuild should happen in a separate module namespace rather than by extending the old guided runtime code. That boundary is defined in `separate_dynamic_module_rebuild.md`.

## Core Idea

The system separates:
- `lesson frame`: pre-generated, bounded, grounded
- `lesson delivery`: dynamic, conversational, adaptive inside constraints

The learner should experience the lesson as a live tutoring session. Under the hood, the runtime is controlled, measurable, and versioned.

This runtime should be generated by a separate pipeline and stored in a separate frame format. It should not be treated as another rendering of the existing block-based lesson JSON.

The upstream planning backbone can still reuse strong syllabus extraction and blueprint planning, but the guided workflow must remain product-level separate from the old static lesson workflow. In practice this now means:
- old/static authoring remains in `/admin/module`
- guided authoring runs in guided-only planner routes such as `/2365/admin/guided-module` and `/gcse/science/biology/admin/guided-module`
- the guided learner runtime remains `/guided-chunk/[lessonCode]`

Current reset decision:
- generated guided content is not the trusted baseline right now
- the guided platform has been reset back to a built-in frame baseline
- one clean stored lesson is the current proof point
- generation should be rebuilt on top of that simpler baseline later
- the next runtime should start from minimal constraint and only add rails when real failures justify them

## Runtime Model

The target runtime is `guided_chunk_v1`.

Core characteristics:
- one shared tutor/learner session window
- one short teaching chunk at a time
- one immediate check after teaching
- pre-generated first questions, then bounded live follow-ups
- short repair when needed
- optional microbreak exercise at planned boundaries
- inline image or diagram injection when instructionally useful
- pseudo-streamed tutor turns so pre-generated content feels live
- progression based on observed learner performance

The system is not a freeform chatbot and not a fully live lesson generator.

The stored lesson frame is the current source of truth.
The runtime must display:
- `loSequence` in stored order
- then `chunkPlan` in stored order

If the lesson feels wrong, inspect the stored frame first before assuming a runtime ordering problem.

It is also not just a draft artifact generator anymore. Guided lessons now have a governed content lifecycle:
- `draft`
- `needs_review`
- `approved`
- `published`
- `retired`

The learner runtime should resolve published guided content only. Drafts and approved-but-unpublished versions are previewed through guided admin and guided planner tooling.

## Static vs Dynamic

### Static
- curriculum grounding
- lesson scope
- chunk sequence
- teaching core per chunk
- vocab pack
- first recall questions
- candidate deeper questions
- misconception and repair patterns
- microbreak exercise slots
- asset injection points

### Dynamic
- tutor phrasing
- exact follow-up question from bounded choices
- whether to repair or advance
- pacing and reveal behavior
- support level within the runtime rules
- transition language

### Generator Memory
- chunk generation should be sequential inside each LO
- prior chunks belong in generation history, not the system prompt
- the generator should also maintain structured memory:
  - covered concepts
  - introduced vocab
  - handled misconceptions
  - chunk summaries

This is the main mechanism for reducing overlap and improving continuity across chunks.

## Why This Is Better Than A Fixed Runtime

The new model should:
- reduce cognitive overload by segmenting instruction
- increase active recall frequency
- make the lesson feel more tutor-like
- allow short repair loops before errors harden
- make fatigue and dropout more measurable
- create a better surface for measured product improvement

## Dual Measurement Model

The platform measures two different systems.

### Learner-performance measurement
Used to adapt instruction for the learner.

Questions:
- what do they know?
- what do they keep getting wrong?
- what support do they need?
- when should they get repair, retry, or review?

### App-performance measurement
Used to improve the product itself.

Questions:
- which runtime design performs better?
- which chunk patterns cause fatigue?
- which repair patterns work?
- which layout or pacing changes improve retention?

A learner failure is not automatically an app failure. App-level redesign must be based on aggregate patterns and outcomes.

## AI In The System

AI operates in two loops.

### Learner loop
The AI adapts instruction inside the existing framework.
Examples:
- repair after wrong answer
- bounded support adjustment
- spaced review triggers

### Product loop
The AI analyzes aggregated data and proposes or applies bounded product changes.
Examples:
- prompt refinements
- chunk-length adjustments
- repair-pattern changes
- microbreak timing changes
- layout or pacing changes

This is a controlled improvement system, not unrestricted self-modifying software.

## V1 Boundaries

V1 is deliberately narrow.

It must prove:
- learners can use the shared tutor-thread runtime without increased confusion
- chunk-by-chunk delivery is viable
- repair and progression work
- telemetry produces decision-useful evidence
- at least one adaptive move measurably helps

It does not need to prove:
- full personalization
- full AI-autonomous redesign
- optimal behavior for every learner type
- replacement of every current lesson surface

Current practical narrowing:
- do not optimize the generator first
- do not engineer every edge case up front
- get one authored guided lesson feeling good
- then rebuild generation and automation around that

## What Makes The Platform Research-Led

The system is based on evidence for:
- segmentation and chunking
- retrieval practice
- guided instruction for novices
- explicit misconception repair
- bounded conversational tutoring
- spaced review
- just-in-time vocabulary support

The product should never optimize only for vibe, novelty, or engagement. Learning outcomes, repair quality, fatigue markers, and retention remain the primary criteria.

## What Makes The Platform Operationally Strong

The platform is designed so that:
- every important learner event is measurable
- every runtime variant is versioned
- every shipped change is traceable
- every analysis cycle can lead to measured improvements or rollback

This now applies to guided content generation as well:
- generated guided versions are persisted
- grading reports are persisted with each version
- the admin surface exposes the report so recurring generator failures can be tightened systematically
- old versions are retained for before/after comparison rather than overwritten

This matters as much as the tutoring behavior itself.

There is also a temporary POC inspection rule:
- full guided runtime payloads and conversation history should be inspectable in console logs
- browser console and server console are now part of the debugging workflow
